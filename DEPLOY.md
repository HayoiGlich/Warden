# Развёртывание: разделение коллектора и веб-«управлялки»

## Идея

- **Коллекторы** (их будет несколько) остаются **нативными на Windows**-серверах
  (например, `av-sv-event`). Каждый коллектор:
  - читает журналы (Windows Event Forwarding) и пишет события в **свой**
    PostgreSQL (таблица `logins`);
  - запускается как служба / задача планировщика (`backend/modules/pars_to_db.py`).
  - **Мы его не трогаем** при разделении.

- **Веб-«управлялка»** (Winlog-поиск + AD-админка + Zabbix-дашборд) выносится на
  отдельный хост в **Docker**. Она только **читает** БД коллекторов по сети,
  ничего в них не пишет.

```
[av-sv-event]  Windows: WEF + импортёр  ->  PostgreSQL #1  \
[collector-2]  Windows: WEF + импортёр  ->  PostgreSQL #2   >--(чтение по сети)--> [Docker] веб-«управлялка»
[collector-N]  ...                      ->  PostgreSQL #N  /
```

## Реестр коллекторов (несколько БД)

Веб-часть опрашивает все коллекторы параллельно (fan-out), помечает каждое
событие именем коллектора и сливает результат. Источник списка (по приоритету):

1. `COLLECTORS_JSON` — встроенный JSON в переменной окружения;
2. файл `collectors.json` (путь — `COLLECTORS_FILE`, по умолчанию `./collectors.json`);
3. **fallback** — один коллектор из `DB_*` в `.env` (текущее поведение).

Пока коллектор один — ничего настраивать не нужно, работает из `.env`.
Когда появятся ещё:

```bash
cp collectors.example.json collectors.json   # заполнить host/port/db/user/password
# раскомментировать volume в docker-compose.yml
```

## Развёртывание коллектора (Windows, на каждый сервер №1/№2/№3)

Импортёр одноразовый (отработал и вышел), поэтому он запускается **Задачей
планировщика**, а не как служба. Параметры читаются из окружения / `.env`
(`backend/modules/pars_to_db.py`), хардкода больше нет.

**1. Скопировать на сервер в одну папку** (например `C:\winlog-collector\`):
- `backend/modules/pars_to_db.py`
- `backend/modules/database.py`  ← обязательно рядом (`from database import ...`)
- `collector/run_importer.bat`
- `collector/register_task.ps1`
- `collector/.env.example` → переименовать в `.env` и заполнить под сервер

**2. Поставить Python и зависимости** (только запись в БД, без Windows-пакетов):
```powershell
pip install -r requirements-collector.txt
```
В `run_importer.bat` укажите путь к `python.exe` (переменная `PYTHON`).

**3. Подготовить локальный PostgreSQL** на сервере:
- импортёр пишет в БД из `.env` (по умолчанию `127.0.0.1/logs`);
- чтобы веб-часть читала его по сети: `listen_addresses='*'`, строка в
  `pg_hba.conf` под IP веб-хоста, firewall 5432.

**4. Убедиться, что WEF складывает события** в `EVTX_PATH`
(по умолчанию `D:\ForwardedEvents\ForwardedEvents.evtx`).

**5. Зарегистрировать задачу** (PowerShell от администратора):
```powershell
cd C:\winlog-collector
.\register_task.ps1 -IntervalMinutes 5
```
Задача идёт под `SYSTEM` с наивысшими правами — `Get-WinEvent` и доступ к evtx
работают без UAC-промпта.

**6. Добавить сервер в реестр коллекторов** веб-части — запись в `collectors.json`
(`host` = IP сервера, не `127.0.0.1`).

> ⚠️ Сверьте параметры БД: импортёр и запись в `collectors.json` должны
> указывать на один и тот же PostgreSQL с одинаковыми кредами.

## Запуск веб-части в Docker (Linux, полностью в контейнере)

Образ **самодостаточный**: внутри Python, все зависимости и собранный React.
На Linux-сервере НЕ нужны ни исходники, ни venv, ни Node — только:

| Файл на хосте | Зачем |
|---|---|
| `docker-compose.yml` | как запускать |
| `.env` | секреты/настройки (`DB_*`, `AD_*`) — подаётся в контейнер, в образ не зашит |
| `collectors.json` | список коллекторов (если их >1); монтируется в контейнер |

Своей БД у сервиса нет — он только читает Postgres коллекторов по сети, поэтому
**Postgres в compose не нужен**.

### Вариант 1 — собрать образ на самом сервере (просто)
```bash
# на Linux-сервере: рядом docker-compose.yml, .env, collectors.json
docker compose up -d --build
docker compose logs -f web
# UI: http://<host>:8000/   (/winlog, /ad-users)
```
Исходники нужны только в момент `build`; дальше всё живёт в образе.

### Вариант 2 — без исходников на сервере (через реестр)
Собрать образ где угодно и положить в registry, на сервере держать только
`docker-compose.yml` + `.env` (+ `collectors.json`):
```bash
# где есть исходники:
docker build -t <registry>/winlog-web:1.0 .
docker push <registry>/winlog-web:1.0
```
В `docker-compose.yml` заменить `build: .` на `image: <registry>/winlog-web:1.0`
и на сервере выполнить `docker compose up -d`.

Особенности образа: multi-stage (сборка React → рантайм Python), запуск под
непривилегированным пользователем `appuser`, `HEALTHCHECK` дёргает `/api/health`.
Windows-зависимости коллектора (`pywin32`, `python-evtx`) в образ **не входят**
(см. `requirements-web.txt`).

> Если коллектор один и `collectors.json` не нужен — закомментируйте `volumes`
> в `docker-compose.yml` (иначе Docker создаст пустой каталог вместо файла), и
> сервис возьмёт единственный коллектор из `DB_*` в `.env`.

> **Мониторинг (Zabbix) вынесен** в отдельный сервис — папка `monitoring/`
> (свой Docker, свой сервер). В этом приложении мониторинга больше нет.

## Авторизация

Вход в приложение защищён формой логина (cookie-сессии). Два источника учёток:
- **Локальный админ** — по умолчанию `admin` / `admin1`; при первом входе
  **обязательна смена пароля**. Хранится хэшом (pbkdf2) в App-БД.
- **AD-пользователи** — по доменному логину/паролю (LDAP-bind к `AD_SERVER`).
  Если задан `AD_LOGIN_GROUP`, пускаются только участники этой группы.

**App-БД** (учётки + настройки) — отдельный Postgres, сервис `db` в
`docker-compose.yml`, данные в volume `appdb-data` (переживает пересоздание
контейнеров). Это своя база приложения, не связана с базами коллекторов.

Обязательные переменные в `.env`:
- `SESSION_SECRET` — длинная случайная строка (подпись cookie). **Смените в проде!**
- `APP_DB_PASSWORD` — пароль App-БД (одинаковый для сервисов `web` и `db`).

> Полностью «без локальных файлов» теперь означает: на хосте только
> `docker-compose.yml` + `.env` (+ `collectors.json`); данные App-БД живут в
> docker-volume, а не в файлах рядом.

## Сетевые требования к хосту веб-части

С контейнера должны быть доступны:

| Назначение | Куда | Зачем |
|---|---|---|
| PostgreSQL коллектора(ов) | `host:5432` каждого | Winlog-поиск (чтение) |
| LDAPS на DC | IP DC `:636` | AD-админка + профили в Winlog |

Если имена резолвятся только в домене — добавьте записи в `extra_hosts`
(см. `docker-compose.yml`).
